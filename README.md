## Key Operations Performed

This assignment addresses 15 questions spanning theoretical concepts and practical PySpark implementations.

### Theoretical Concepts

- **Spark vs. MapReduce** — How in-memory DAG execution outperforms disk-reliant MapReduce for iterative workloads
- **Wide vs. Narrow Transformations** — Understanding when a shuffle is triggered (e.g., `groupBy`, `join`) vs. when data stays partition-local (e.g., `filter`, `map`)
- **DataFrame Immutability** — How Spark tracks state changes through lineage rather than mutation
- **Schema Inference Risks** — Why `inferSchema=True` can misclassify types with messy or mixed-format data

### Practical PySpark Implementations

**Data Cleaning**
- Removing duplicate rows using `dropDuplicates()` with column subsets
- Handling missing values via `na.drop()` and `na.fill()`
- Filtering invalid records: null emails, empty usernames

**Transformations**
- Schema manipulation and type casting with `cast(TimestampType())`
- Renaming columns using `withColumnRenamed()` and `alias()`

**Aggregations & Filtering**
- Conditional filtering with `.filter()` and `col().between()`
- Grouping by dimensions: `store_id`, `city`, `product_category`
- Multi-statistic aggregations: `min`, `max`, `avg`, `sum`

## Key Insights

- Null and empty-string handling must be treated separately in PySpark — `na.drop()` does not catch empty strings `""`
- `inferSchema=True` misread mixed-format timestamp columns as plain strings; explicit casting was required
- Wide transformations (`groupBy`, `distinct`) triggered shuffles visible in the Spark UI DAG, confirming the performance difference vs. narrow operations
- Using `dropDuplicates(subset=[...])` is significantly more targeted than full-row deduplication, preserving valid records that share partial key overlap
